3. 配置应用回滚

  • 当我们在部署应用的时候因功能或运行错误导致环境不可用或不是我们所期望的，进行回滚很有必要。回滚我们可以通过在页面触发临时回滚，也可以添加自动回滚阶段到流水线。
  • 使用Deployment对象时，可以在“群集”选项卡中看到历史记录：

    1. 临时回滚
      选中ServerGroup -> Deployments actions -> Undo Rollout -> 选择版本

    1. 自动回滚
      添加阶段 -> Undo Rollout(Mainfest)

  • 实例：人工确认后执行回滚

1. 管道表达式

概述

  • 管道表达式可以在管道执行期间进行参数动态设置和访问变量。
  • 可以在Spinnaker管道阶段的几乎所有文本字段中使用表达式。
  • 管道表达式可帮助您在管道执行过程中使用有关系统状态的任意值。
  • 您可以使用它们来打开或关闭管道的特定阶段，动态命名堆栈，检查另一阶段的状态以及执行其他操作。
  • ${expression here}
    ^: 注意：管道表达式语法基于SpringExpression Language（SpEL）

管道工具-Functions - 函数

  • 在管道表达式中输入 “#”号，可以显示所有可用的辅助函数。

管道工具-Properrties-属性

  • 辅助属性是变量，可以引用有关当前管道执行的全局信息。
    • execution 引用当前的管道执行信息。
    • trigger 访问有关管道触发器的信息。例如，如果您已经设置了Jenkins触发器，并且想知哪个构建触发了当前管道，则可以使用表达式访问该信息${trigger[“buildInfo”][“number”]}。
  • 要快速列出可用的辅助属性，可以在管道表达式中输入问号。显示列表后，您可以开始键入以缩小列表的范围，或按Enter键将选定的属性添加到表达式中。

管道工具-Property file-属性文件

  • 将Jenkins作业属性传递到管道。文件类型可以是YAML或者JSON

2. 蓝绿部署

  • Kubernetes本身不支持蓝/绿部署策略。要实现必须使用不同的名称创建一个新的Deployment，然后管理外部的Service流量。手动删除较旧的部署。

Spinnaker蓝绿部署

  • 需要一个Service，该Service将使用与副本集选择器不匹配的选择器将流量路由到Pod。
  • 基于副本集（ReplicaSet）部署，以使生命周期易于管理。
  • 使用注释（annotations）流量管理自动化。

实验-1.创建LB（Service）

  • Service选择器与副本集容器的标签不匹配。这很重要，因为Spinnaker禁用副本集时，它会通过从副本集中删除服务的选择器来将副本集从服务的流量路径中删除。

金丝雀部署

什么是金丝雀部署

金丝雀部署是将发行版发布到一部分用户或服务器的一种模式。这个想法是首先将更改部署到一小部分服务器，进行测试，然后将更改推广到其余服务器。一旦通过所有运行状况检查，当没有问题时，所有的客户将被路由到该应用程序的新版本，而旧版本将被删除。

  • 1.部署Prometheus(对Kubernetes集群监控)
  • 2.开启Spinnaker对canary支持
  • 3.创建Canary Config文件（指定指标数据）
  • 4.配置金丝雀流水线，添加金丝雀分析步骤。

实验-3.6.1 流水线实现-金丝雀分析阶段

  • Analysis Type: 分析类型，Real Time 实时分析（当前阶段开始），Retrospective 指定分析时间段。
  • ConfigName: 指定金丝雀配置文件的名称。 Lifetime: 指定分析数据的总运行时间。
  • Delay： 延迟时间，多少分钟后开始运行分析。 Interval: 间隔时间。 Step: 时间序列的间隔。
  • Baseline Offset ：从金丝雀分析开始以来基线数据收集的分钟数偏移。对于比较相对标准前的指标很有用。
  • LookBackType：分析类型。
  • Growing在分析过程中将考虑金丝雀的整个持续时间。
  • Sliding金丝雀将为其分析报告使用最新的指定分钟数（对于跨多天的长时间运行的金丝雀很有用）。

实验-3.6.2 流水线实现-金丝雀分析阶段

  • 填写基线版本和金丝雀版本的名称和所在的名称空间。
  • 在配置金丝雀配置文件的时候添加的过滤模板中使用的scope参数对应的是Baseline和Canary的名称。
  • 设置分数范围：得分小于50失败，大于等于80通过。
